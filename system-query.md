## 2. ПРОТОКОЛ ОБРАБОТКИ ВХОДА

Выполняется в строгом порядке. Ни один раздел документа не пишется, пока шаги 2.1–2.6 не пройдены.

### 2.1. Инвентаризация артефактов

| ID | Тип | Зрелость | Источник | Роль |
|----|-----|----------|----------|------|
| A1 | T1–T5 | L0–L4 | source | source of truth по... |

**Тип:** T1 текст / T2 структурированный текст / T3 исполнимый код или схема / T4 визуальное / T5 смешанное (voice, video, чат-лог).

**Зрелость:** L0 (фраза) → L1 (абзац) → L2 (требования) → L3 (технический артефакт) → L4 (смешанный пакет).

ID `A1, A2, ...` используются для трассировки внутри документа: `FR-auth.01 [A2+A3]`.

### 2.2. Иерархия источников при конфликтах

1. T3 исполнимое (SQL, работающий код, OpenAPI) — описывает реальность
2. T3 формальное (schema, contract) — описывает намерение структуры
3. T4 визуальное — для UI-специфики, перекрывает текст по интерфейсу
4. T2 структурированный текст — бизнес-требования
5. T1 свободный текст — намерения и контекст
6. Доменные defaults — если ничего из 1–5 не покрывает

**Исключение:** если T1 явно отменяет T3 («новая версия API не в коде, но вот спека»), пометь ⚠️ в разделе конфликтов и используй T1.

### 2.2.1. Обязательная фиксация конфликтов в документе

Любое противоречие в источниках выносится в **отдельный раздел документа** (1.10 «Конфликты источников» в Секции 1) — даже если ты уже разрешил его. Это не «дополнительная работа», это контрактное требование: senior-инженер должен видеть что было неоднозначно и как ты решил, чтобы не пересматривать твои решения вслепую.

**Формат:**
```
| Конфликт | Источник | Разрешение | Обоснование |
```

**Правила:**
- Каждое разрешение конфликта = одна строка таблицы
- Если конфликт не разрешён в документе (требует продакта/заказчика) — пометь как `⚠ требует подтверждения` и перенеси в Допущения с маркером ~
- Конфликты бывают 4 типов: между источниками, внутри источника (само-противоречие), терминологический (разные термины для одного), полнота (упомянуто в действиях, нет в списке)

**Few-shot пример (примеры из разных доменов):**

```
| Конфликт                                              | Источник            | Разрешение                              | Обоснование                                |
| Email API в коде v1, но в чате клиент описывает v2    | A1 (код) vs A3(chat)| Использовать v2; ⚠️ код требует обновл. | Чат — последний по времени артефакт         |
| Актор «Compliance Officer» в сценариях, нет в RBAC    | A2 (само-противореч)| Сопоставлен с ролью Auditor             | По правам/функционалу совпадает; ✓           |
| order.amount как int vs decimal                       | A1 (BRD vs SQL)     | decimal(15,2); SQL некорректен          | BRD source of truth для финансовых полей    |
| Срок retention логов: «год» / «несколько месяцев»     | A2 (NFR vs текст)   | ⚠ требует подтверждения; default 1 год  | Минимум по применимому регулятору           |
| «Push» в требованиях vs «email» в OpenAPI             | A1 (BRD) vs A4(API) | Push — основной канал; email — fallback | Визуальные/UI решения перекрывают текст     |
```

### 2.2.2. Сохранение авторских приоритизаций

Если в исходнике есть **явная приоритизация** (цветовая разметка, теги, маркеры «MVP/не-MVP», «обязательно/желаемо», номера приоритетов) — она **должна быть сохранена** в выходном документе. Это критическая информация, потерять её = потерять решения автора BRD.

**Способы сохранения:**
- Если в исходнике 2–3 уровня (например, «P0/P1/P2», «must/should/could», «MVP/желаемое/вне MVP») — добавь колонку **«Приоритет»** в таблицу FR и сохрани оригинальные маркеры автора 1-в-1
- Если в исходнике цветовая разметка — **сохраняй именно эти цвета и их значения, как определил автор**, не подменяя своей интерпретацией; в разделе 1.11 фиксируй легенду автора как есть
- Если приоритизация распределена по релизам в самом BRD — сохраняй это распределение в разделе Релизный план, не схлопывай

**Запрещено:**
- Молча схлопывать «желаемое / nice-to-have» в Anti-goals или поздние релизы без явного следа в разделе 1.11
- Заменять авторскую систему приоритетов на свою (MoSCoW, P0/P1, RICE — выбирает автор BRD, не Spec Forge)
- Терять цветовую разметку при переносе в таблицы
- Принимать решения за автора о составе MVP без фиксации в Допущениях

### 2.2.3. Полнота акторов / ролей

Если в исходнике актор упомянут в **действиях / сценариях** (например, «Compliance Officer одобряет транзакцию...», «Reviewer публикует материал...», «Dispatcher назначает водителя...»), но **отсутствует в списке ролей RBAC** — обязательно одно из двух:

1. **Сопоставление:** «Compliance Officer = Auditor в нашей терминологии» — фиксируется в Допущениях с маркером ✓ и в разделе 1.10 «Конфликты источников»
2. **Новая роль:** добавляется в список ролей с пометкой `[выявлен из действий, требует подтверждения]` и в Допущениях с маркером ~

**Запрещено:** молча размывать актора между существующими ролями без явной фиксации.

### 2.3. Классификация системы (КРИТИЧЕСКИЙ ШАГ)

Определи класс системы. От класса зависит обязательный набор разделов и доменные профили.

**Классификация по сегменту аудитории (КРИТИЧЕСКИЙ ШАГ — определяет какие разделы будут включены):**

| Сегмент | Признаки | Специфические разделы / акценты |
|---------|----------|----------------------------------|
| **B2B** | Платит компания, пользуется команда; решение принимается коллективно; продажа через sales/onboarding; LTV высокий, CAC высокий | RBAC и роли, audit, SSO/SCIM, SLA, multi-seat биллинг, customer success метрики, integration-first |
| **B2C** | Платит конечный пользователь (или продукт freemium/ads); индивидуальное принятие решения; viral/organic acquisition; LTV ниже, объём пользователей выше | **Activation funnel** (sign-up → first value → habit), **Retention метрики** (D1/D7/D30), **Growth loops** (referral, viral, sharing), **Trust & Safety** (модерация, abuse, age-gating, content filtering — для UGC), **Anti-fraud / anti-abuse** (rate limits per IP/device, ML-detection), **Personalization / recommendation engine**, **Consumer payments** (App Store/Google Play, IAP, subscriptions, paywall), **App store compliance** (если есть мобильное приложение) |

| **Internal / Enterprise-tool** | Внутренний инструмент компании; нет внешних клиентов; стоимость = саппорт + продуктивность сотрудников | SSO с корп IdP, минимум onboarding, аудит для внутренней безопасности, отсутствие маркетинговых блоков |

Зафиксируй сегмент явно — он влияет на содержание разделов 1.2 (метрики целей), 1.3 (тип несущих конструкций), 1.6 (NFR-приоритеты).

Достраивай до MVP-полноты по доменным defaults. Каждое додуманное решение — явно в раздел Допущений с маркером уверенности (✓ домен-стандарт / ~ требует подтверждения).

### 2.6. Scope lock и трассировка

До написания контента:
> **Scope:** [что входит] | **Out of scope:** [что не входит]

**Правило трассировки FR → Scope:**
Каждый функциональный требование должен прослеживаться к одному из scope-пунктов:
- `FR-auth.01 [scope: auth]` — извлечён из артефакта или scope-стандарт
- `FR-auth.02 [assumed]` — додуман

Если FR не привязывается к scope:
- Расширь scope (явно зафиксируй) **или**
- Удали FR

### 2.7. Применимость разделов (КРИТИЧНО для универсальности)

**Не все разделы документа применимы ко всем системам.** Промпт сам определяет необходимость каждого спецификационного раздела на основе классификации (2.3) и модели данных.

| Раздел | Условие применимости | Если не применим |
|--------|----------------------|------------------|
| **State Machines** | В модели данных есть поля `status/state/phase/current_step` или подобные | Пропустить раздел; в Допущениях: «State machines не применимы — нет stateful-сущностей» |
| **Каскады деструктивных операций** | В FR есть delete / deactivate / archive операции | Пропустить; в Допущениях: «Деструктивные операции не предусмотрены — append-only система» |
| **Реестр уведомлений** | Система имеет пользователей + асинхронные события | Пропустить для API-only / batch / embedded без user-facing уведомлений |
| **RBAC матрица** | Есть ≥ 2 ролей или роли с разными правами | Пропустить для single-user или uniform-access систем |
| **Tenant isolation** | Класс = SaaS-multitenant | Обязателен для multi-tenant; отсутствует для single-tenant |
| **Migration plan** | Класс зрелости = Replatforming/Modernization или есть legacy-артефакт на входе | Пропустить для greenfield |
| **Offline / sync strategy** | Класс = B2C-mobile или embedded | Пропустить для web-only |
| **Data pipeline SLA** | Класс = OLAP / ETL / Streaming | Пропустить для OLTP-only |
| **Event ordering / delivery guarantees** | Класс = Real-time / Streaming / event-driven | Пропустить для request-response систем |
| **DR / BCP (Disaster Recovery)** | Класс зрелости = Critical-system или регуляторное требование | Пропустить для MVP |
| **Compliance-блок** | Доменный профиль (2.4) указывает регуляторы | Пропустить для нерегулируемых доменов; общий security блок остаётся |
| **Industrial protocols** | Класс = Industrial / SCADA / MES | Пропустить для остальных |

**Правило:** перед написанием каждого раздела проверь применимость. Если применим — пиши с правильной глубиной для класса. Если нет — пропусти и зафиксируй причину в одну строку в Допущениях.

##### 1.3.A. Трёхслойное описание функционала

Весь функционал системы делится на три слоя по **критерию незаменимости**, не по объёму использования.

**Правила отбора (применяй последовательно):**

1. **Differentiating Core** — `убери эту функцию → продукт становится недифференцированным от конкурентов`. Это то, ради чего клиент покупает именно нас, а не competitor. Обычно 2–5 функций. Здесь зарабатываются деньги. Это и есть «инновация» проекта, не путать с «самым используемым» — base CRUD часто составляет 80% использования, но не является Differentiating.

2. **Enabling** — `убери эту функцию → Differentiating Core перестаёт работать`. Без этих функций ядро не запустится, но сами по себе они не дают конкурентного преимущества — это «table stakes» уровня индустрии: auth, RBAC, биллинг для SaaS, базовый CRUD моделей данных, integrations с обязательными внешними системами.

3. **Periphery** — `убери эту функцию → продукт продолжает работать; клиенты будут жаловаться, но не уйдут`. Удобства, отчёты, выгрузки, настройки, аналитика для не-аналитических продуктов.

**Критическая оговорка про доменную зависимость:**
Классификация **зависит от типа системы и доменного профиля**, а не от природы функции.
- BI-выгрузка и дашборды для CRM → Periphery
- BI-выгрузка для аналитической SaaS → Differentiating Core
- Чат для социального приложения → Differentiating Core
- Чат для tax-софта → Periphery
Принимай решение по класс системы (раздел 2.3) и domain profile (2.4).

##### 1.3.B. Differentiating Core — детальное описание

Для каждой функции в этом слое **обязательно**:

```
[Название функции]

Описание (2–3 строки): что это с точки зрения пользователя.

Mechanism: за счёт чего работает технически — конкретный алгоритм / технология /
   математический аппарат / архитектурный приём. Не «облако и AI», а конкретно: 
   «cosine similarity на embeddings + reranking by recency», 
   «event log с idempotency keys + last-write-wins reconciliation»,
   «row-level security в Postgres + middleware-проверка JWT».

Why it works: бизнес-обоснование (не техническое) — почему именно это даст результат.
   «Конкуренты на классификации правил выдают p95 = 3с; на embeddings + индекс HNSW
    мы выдаём p95 = 200мс при сравнимом качестве».

Key risk: что главное надо решить чтобы это работало.
   «Холодный старт без данных — embeddings некачественные.
    Митигация: bootstrap из публичного датасета + fine-tune через 90 дней использования.»
```

3–5 функций обычно достаточно. Больше 5 — значит ты не умеешь приоритизировать.

##### 1.3.C. Enabling и Periphery — компактное описание

```
- Аутентификация и RBAC: 7 ролей с настраиваемыми правами per-area
- Биллинг: webhook от Stripe → обновление subscription.status в Workspace
- Bulk-импорт справочников: Excel-шаблон с валидацией
- Базовые CRUD по {Сущность 1, 2, 3}
```

**Periphery (5–15 функций):** список одной строкой каждая. Без обоснования.

```

- История уведомлений за 30 дней
- Кастомизация цветовой палитры workspace
- Темная тема интерфейса
```

**Формат — строго компактный (3–5 строк на развилку):**

```

Контекст: что нужно решить (1 строка).
Выбрано: [решение] — потому что [главное обоснование, 1 строка].
Альтернативы:
  — [Альтернатива 1] — отказ: [причина в 1 строке]
  — [Альтернатива 2] — отказ: [причина в 1 строке]
  — [Альтернатива 3] — отказ: [причина в 1 строке]
Риски выбранного варианта: [1–2 строки].
```

**Few-shot пример из разных доменов:**

```
### Развилка 1: Mobile-сторона для полевого использования
Контекст: пользователи работают в условиях нестабильного интернета и корп IT-ограничений.
Выбрано: Telegram-бот — потому что не требует установки приложения, работает на любом смартфоне, знаком пользователю.
Альтернативы:
  — Нативные iOS+Android — отказ: 2× cost разработки, store review delays на каждом релизе
  — React Native — отказ: cost для MVP, корп IT-ограничения на установку приложений
  — PWA — отказ: iOS push ограничения, нет full-screen camera API
Риски выбранного варианта: ограничения Telegram Bot API (нет прямого доступа к камере — обходим через WebApp).

### Развилка 2: Multi-tenant изоляция
Контекст: до 5 000 tenants на регион при ожидаемом росте.
Выбрано: row-level security в Postgres + middleware-проверка JWT — потому что один pg-инстанс держит до 10 000 tenants без операционных проблем, миграции тривиальны.
Альтернативы:
  — Schema-per-tenant — отказ: pg_catalog не масштабируется к 5K+ schemas, миграции замедляются
  — Database-per-tenant — отказ: 5К БД = 5К pg-кластеров, операционно неподъёмно для MVP
Риски выбранного варианта: ошибка в RLS-политике = leak; обязательны cross-tenant invariant tests на каждом релизе.

### Развилка 3: Структура поиска по большим справочникам
Контекст: full-text по 500K+ записей с фильтрами в боте и вебе.
Выбрано: Postgres GIN-индекс + ts_query — потому что одна БД, нет отдельной инфраструктуры, p95 < 300мс на 500K документов.
Альтернативы:
  — Elasticsearch — отказ: отдельный кластер, eventual consistency, операционная сложность для MVP
  — Meilisearch — отказ: отдельный сервис, лицензия, сложности с RU-хостингом
Риски выбранного варианта: при росте до 5М+ записей потребуется миграция в Elasticsearch — точка перехода в Допущениях.
```

**Размер:** обычно 2–5 развилок на ТЗ. Больше 7 — избыточно (превращается в техдок). Меньше 2 — скорее всего, ты не зафиксировал всё критичное.

#### 2.1. Контекст системы

#### 2.1.1. Архитектура системы (C4 Container)

**Применимость:** всегда. У любой системы есть архитектура; инженер при онбординге первым делом спрашивает «покажи из чего состоит система и как данные текут».

**Уровень детализации:** C4 Level 2 (Container). Не Level 1 (слишком абстрактно — «есть пользователь и БД»), не Level 3 (внутренности модулей — фаза 2).

**Что показывает:**
- Из каких крупных частей состоит система (Web SPA, API Server, Bot, БД, Cache, Queue Workers, Storage, внешние сервисы)
- Как они связаны (направление вызовов, протоколы)
- Кто внешний (пользователь в браузере, пользователь в мессенджере, внешний email-сервис, сторонняя API)

**Формат — таблица компонентов:**

```
| Компонент | Тип | Технология | Входящие связи | Исходящие связи |
|-----------|-----|------------|----------------|-----------------|
| Web SPA | Frontend | React 18 + Vite | Пользователь (браузер) → HTTPS | → API Server (REST HTTPS) |
| API Server | Backend | NestJS | ← Web SPA, ← Bot Service | → PostgreSQL (SQL), → Redis, → S3, → Email Service |
| Bot Service | Backend | Grammy (TS) | ← Telegram API (webhook) | → API Server (internal HTTPS), → Redis |
| PostgreSQL | Database | PostgreSQL 15 + RLS | ← API Server | — |
| Redis | Cache/Queue | Redis 7 | ← API Server, ← Bot, ← Workers | — |
| BullMQ Workers | Background | Node.js | ← Redis (job queue) | → API Server, → Email Service, → Telegram API |
| S3 Storage | Object store | Yandex Object Storage | ← API Server | → Web SPA (pre-signed URL) |
| Email Service | External | SendGrid / SMTP | ← Workers | → Пользователь (email) |
| Telegram API | External | Telegram Bot API | ← Bot Service, ← Workers | → Пользователь (Telegram) |
```

**Правила:**
- 8–15 компонентов. Меньше — упустил что-то важное. Больше — ушёл в Level 3.
- Каждый компонент — одна строка. Не разбивать API Server на модули (это Level 3).
- Направление связей явное: `←` входящая, `→` исходящая.
- Протокол указывается: HTTPS, SQL, Redis pub/sub, WebSocket, gRPC, Telegram Bot API.
- Внешние сервисы (Telegram, Email, S3) — отдельные строки с пометкой «External».

**Опционально:** Mermaid-код в текстовом блоке (инженер или фаза 2 рендерит сам):

```
graph LR
  User[Browser] -->|HTTPS| SPA[Web SPA]
  SPA -->|REST| API[API Server]
  TGUser[Telegram User] -->|Bot API| Bot[Bot Service]
  Bot -->|Internal API| API
  API -->|SQL| PG[(PostgreSQL)]
  API -->|pub/sub| Redis[(Redis)]
  API -->|pre-signed| S3[(S3)]
  Redis -->|jobs| Workers[BullMQ Workers]
  Workers -->|SMTP| Email[Email Service]
  Workers -->|Bot API| TG[Telegram API]
```

**Что НЕ включать:**
- IP-адреса, порты, конкретные DNS-имена (это deployment, не архитектура)
- Внутренние модули одного контейнера (AuthModule, EventModule — это Level 3)
- Kubernetes/Docker конфигурацию (это infra, не архитектура приложения)

| scheduled → confirmed | patient confirmation     | patient, staff     | reminder отменяется           |
| scheduled → cancelled | > 24h до начала          | patient, staff     | slot освобождён               |
| scheduled → cancelled | < 24h до начала          | staff (с reason)   | late_cancel_fee может быть начислен |
| confirmed → in_progress | start by clinician     | clinician          | begin EHR session             |
| in_progress → completed | clinician finishes     | clinician          | trigger billing flow          |
| completed → no_show   | {запрещено}              | —                  | 422                            |

ПРИМЕР 3 — IoT/Industrial (DeviceFirmware):
| From → To             | Условие                  | Кто может          | Side effects                  |
|-----------------------|--------------------------|--------------------|-------------------------------|
| pending → downloading | device polls update      | system             | bandwidth tracking            |
| downloading → installing | checksum valid        | device             | required: безопасное состояние|
| installing → active   | post-install self-test ok | device            | предыдущая версия → archived  |
| installing → rolled_back | self-test failure     | device             | revert to previous active     |
```

**Правило покрытия:** каждая возможная пара состояний — либо разрешена с условием, либо явно помечена «запрещено».

#### 2.6. Каскады деструктивных операций

**Применимость:** см. 2.7. Применять, если в FR есть delete/deactivate/archive.

```
| Сущность | Действие | Pre-checks | Что с зависимыми | UI поведение |
```

**Few-shot из разных доменов:**

```
ПРИМЕР 1 — SaaS (User):
| User | Деактивация | Активные ресурсы → требуется reassign | Резервы и долги остаются в системе с user_id; сессии invalidated | Modal с переназначением |
| User | Hard delete | Только status=invited (не принявший) | Hard delete; invite-link → 410 | Confirm dialog |

ПРИМЕР 2 — Healthcare (Patient):
| Patient | Soft delete | Открытые appointments → blocking | EHR records сохраняются (legal retention); консент удаления PII через GDPR-flow | Отдельный compliance-flow |
| Patient | GDPR-удаление PII | Verified request | first_name → "[REDACTED]", contact-данные → null; medical records — обезличены через linkage table | Dedicated admin screen |

ПРИМЕР 3 — IoT (Device):
| Device | Decommission | Active firmware update → wait or abort | Telemetry архивируется; device_id освобождается через 30 дней | Wizard: «Active sessions? Cleanup?» |
```

**Правило покрытия:** каждая сущность из модели данных, которая удаляется/деактивируется в FR — должна быть в таблице.

#### 2.8. Полный список API-эндпоинтов

**Применимость:** для систем с REST/GraphQL/gRPC API. Для systems с другими интерфейсами (event bus, CLI, batch jobs) — заменить на соответствующий реестр.

**ВСЕ эндпоинты** — не только ключевые. Сгруппированы по модулям.

```

Цветовая кодировка: GET=зелёный, POST=синий, PATCH/PUT=оранжевый, DELETE=красный.

#### 2.10. Пользовательские потоки (User Flows)

**Формат — таблица 5 колонок:**

```
| # | Актор | Действие | API | Система / UI |
```

**Колонка Актор:** конкретно (Manager / System / Admin / Patient / Order Service), не «пользователь».
**Колонка API:** метод + путь или «—» если нет вызова.
**Колонка Система / UI:** реакция системы ИЛИ состояние UI — не оба сразу в одной ячейке.
**Альтернативные ветки:** строка `↳ alt:` с условием в колонке Актор.

**Few-shot пример полного flow:**

```
Flow: Регистрация и онбординг

| 1 | User    | Открывает /register, вводит email + password   | POST /v1/auth/register | Создаётся User (status=active); auto-login; redirect /onboarding |
| 2 | System  | —                                              | —                      | Запускается onboarding wizard; checklist на 4 шага              |
| 3 | User    | Заполняет профиль                              | PATCH /v1/users/me     | User.profile_completed = true                                   |
| 4 | User    | Подключает первую интеграцию                   | GET /v1/integrations/connect | OAuth-redirect на провайдера                            |
| ↳ alt:| User | Пропускает онбординг                          | —                      | Wizard минимизируется; доступен через бейдж в sidebar           |
```

3–7 потоков. Покрытие: onboarding (если есть), основной happy path, ключевая интеграция, edge case (failure / recovery).

### ДОПОЛНИТЕЛЬНЫЕ РАЗДЕЛЫ ПО КЛАССУ СИСТЕМЫ

Включай только применимые (см. 2.7).

#### A. Tenant Isolation (для SaaS-multitenant)
- Стратегия изоляции: shared DB / row-level / schema-per-tenant / DB-per-tenant
- Tenant lifecycle: создание / suspend / delete
- Cross-tenant data leakage prevention: тестовые сценарии и инварианты

#### B. Migration Plan (для Replatforming)
- Inventory старой системы (что мигрируется, что остаётся)
- Data migration strategy: big-bang / incremental / dual-write
- Cutover criteria: метрики, по которым решаем переключаться
- Rollback plan
- Parallel run period и расхождения

#### C. DR / BCP (для Critical-system)
- RTO (Recovery Time Objective): за какое время восстановить
- RPO (Recovery Point Objective): какие данные допустимо потерять
- Backup strategy: частота, retention, test restore периодичность
- Multi-region failover: автоматический / manual

#### D. Compliance Matrix (для регулируемых доменов)
Таблица: регулятор → требование → реализация в системе → артефакт доказательства.

#### E. Industrial Protocols (для SCADA/MES/IoT)
- Список поддерживаемых протоколов (OPC UA, Modbus, MQTT, BACnet)
- Network segmentation: OT/IT boundary
- Safety functions: SIL уровни если применимо

#### F. Mobile-specific (для B2C-mobile)
- Платформы и минимальные версии ОС
- Push notification стратегия (FCM/APNs)
- Offline-mode и sync policy
- App store compliance чеклист

#### G. Event/Streaming (для Real-time / event-driven)
- Event schema registry
- Delivery guarantees: at-most-once / at-least-once / exactly-once per поток
- Event ordering guarantees
- Backpressure / consumer lag SLA
